 # IP의 한계를 넘어서 — 전송 계층

네트워크 계층에서 IP가 패킷을 목적지까지 전달해준다는 건 배웠다. 그런데 한 가지 의문이 생긴다.

> *"IP가 패킷을 보내준다면, 그걸로 충분한 거 아닌가?"*

결론부터 말하면 **충분하지 않다**. IP에는 두 가지 근본적인 한계가 있다.

---

## IP의 두 가지 한계

### ① 신뢰할 수 없는 통신 (Unreliable)

IP는 패킷이 수신지까지 **제대로 전송되었다는 보장을 하지 않는다**. 이를 **최선형 전달(Best Effort Delivery)** 이라고 부른다. 최선을 다하긴 하는데, 도착 여부는 보장 못 한다는 의미다.

### ② 비연결형 통신 (Connectionless)

IP는 패킷을 보내기 전에 송수신 호스트 간 **사전 연결 수립 과정이 없다**. 그냥 수신지를 향해 패킷을 던질 뿐이다.

> **왜 IP는 이렇게 설계됐을까?**

**성능** 때문이다. 모든 패킷이 제대로 전송됐는지 확인하고, 호스트 간 연결을 수립하는 작업은 시간이 걸린다. IP는 그 오버헤드를 포기하고 속도를 택했다.

그럼 신뢰성이 필요한 통신은 어떻게 할까? → 그게 바로 **전송 계층**의 역할이다.

---

## 전송 계층이란?

**전송 계층(Transport Layer)** 은 네트워크 계층과 응용 계층 사이에 위치하며, IP의 한계를 보완해 세 가지 역할을 한다.

```
응용 계층
    ↕
전송 계층  ← 신뢰성 보장 + 연결 수립 + 포트로 애플리케이션 식별
    ↕
네트워크 계층 (IP)
```

전송 계층의 프로토콜은 두 가지다.

| 프로토콜 | 특성 | 한 줄 요약 |
|---------|------|-----------|
| **TCP** | 연결형, 신뢰성 있음 | 느리지만 확실하게 |
| **UDP** | 비연결형, 신뢰성 없음 | 빠르지만 보장 없음 |

---

## 응용 계층과의 연결 다리 — 포트

전송 계층의 또 다른 역할은 **포트(Port)** 를 통해 응용 계층의 애플리케이션을 식별하는 것이다.

패킷은 호스트까지만 도달하면 되는 게 아니다. **실행 중인 특정 애플리케이션까지** 전달되어야 한다. 예를 들어 같은 컴퓨터에서 브라우저와 게임이 동시에 실행 중일 때, 패킷이 어느 앱으로 가야 할지 구분해야 한다.

그 구분자가 바로 **포트 번호**다.

```
IP 주소  → 어느 호스트인가  (호스트 식별)
포트 번호 → 어느 애플리케이션인가 (프로세스 식별)

192.168.0.15 : 8000
└──────────┘   └──┘
  호스트 식별   앱 식별
```

포트 번호는 **16비트**로 표현되며 `0 ~ 65535` 범위를 가진다.

### 포트 번호의 세 종류

| 종류 | 범위 | 설명 |
|------|------|------|
| **잘 알려진 포트 (Well-Known Port)** | 0 ~ 1023 | IANA가 범용 프로토콜에 할당. system port라고도 함 |
| **등록된 포트 (Registered Port)** | 1024 ~ 49151 | 흔히 사용되는 애플리케이션 프로토콜에 할당 |
| **동적 포트 (Dynamic Port)** | 49152 ~ 65535 | 임시/사설 포트. 클라이언트 프로그램(웹 브라우저 등)에 임의 할당 |

자주 마주치는 잘 알려진 포트들:

| 포트 번호 | 프로토콜 |
|---------|---------|
| 20, 21 | FTP |
| 22 | SSH |
| 53 | DNS |
| 67, 68 | DHCP |
| 80 | HTTP |
| 443 | HTTPS |

그리고 개발할 때 자주 보이는 등록된 포트들:

| 포트 번호 | 설명 |
|---------|-----|
| 3306 | MySQL |
| 6379 | Redis |
| 8080 | HTTP 대체 |

---

### 포트 기반 NAT — NPAT

앞서 네트워크 계층에서 **NAT(Network Address Translation)** 를 배웠다. 사설 IP를 공인 IP로 변환하는 기술이었다.

그런데 기본 NAT는 **사설 IP 하나 당 공인 IP 하나**가 필요하다. 가정의 여러 기기가 하나의 공인 IP를 공유하려면 어떻게 할까?

그게 바로 **NPAT(Network Address Port Translation)** 다. **포트 번호까지 함께 활용**해서 하나의 공인 IP를 여러 사설 IP가 공유할 수 있게 한다.

```
[네트워크 내부]                    [네트워크 외부]
192.168.0.5:1025  ──┐
192.168.0.6:1026  ──┤  NAT 변환  ──  1.2.3.4:b200
                    └─────────────   1.2.3.4:b201
```

NAT 변환 테이블에 `공인 IP:포트 ↔ 사설 IP:포트` 매핑을 저장해서 여러 기기가 하나의 공인 IP를 나눠 쓸 수 있다. 우리 집 공유기가 정확히 이 역할을 한다.

> **포트 포워딩(Port Forwarding)이란?**

네트워크 내 특정 호스트에 IP 주소와 포트 번호를 미리 할당하고, 외부에서 특정 포트로 접근했을 때 해당 호스트의 내부 포트로 전달해주는 기능이다.

```
외부에서 1.2.3.4:1234 로 접근
         ↓
내부 192.168.100.100:1025 로 전달
```

공유기 설정에서 흔히 볼 수 있는 그 기능이다.

---

### 좀 더 알아보기 — ICMP

**ICMP(Internet Control Message Protocol)** 는 IP의 신뢰할 수 없는 전송 특성을 **보완**하기 위한 네트워크 계층 프로토콜이다.

두 가지 역할을 한다.

- **오류 보고**: 전송 과정에서 발생한 문제 상황 알림
- **진단 정보 제공**: 네트워크 상태에 대한 정보 제공

| 타입 | 코드 | 설명 |
|------|------|------|
| 수신지 도달 불가 (3) | 0 | 네트워크 도달 불가 |
| | 1 | 호스트 도달 불가 |
| | 2 | 프로토콜 도달 불가 |
| | 3 | 포트 도달 불가 |
| 시간 초과 (11) | 0 | TTL 만료 |
| 에코 요청 (8) | 0 | 에코 요청 |
| 에코 응답 (0) | 0 | 에코 요청에 대한 응답 |

우리가 자주 쓰는 `ping` 명령어가 바로 ICMP의 **에코 요청/응답** 메시지를 기반으로 동작한다.

> **주의**: ICMP는 IP를 신뢰할 수 있게 만드는 게 아니라, 신뢰할 수 없는 특성을 보완하는 **도우미** 역할이다. 데이터그램 전달을 보장하지는 않는다.

## TCP와 UDP

### TCP란?

**TCP(Transmission Control Protocol)** 는 **신뢰할 수 있는 연결형 프로토콜**이다.

통신을 시작하고 끝맺는 과정이 있고, 그 과정에서 오류 제어, 흐름 제어, 혼잡 제어 등 다양한 기능을 제공한다. TCP 통신은 세 단계로 이루어진다.

```
① 연결 수립
      ↓
② 데이터 송수신  ← 오류 제어, 흐름 제어, 혼잡 제어
      ↓
③ 연결 종료
```

---

### TCP 세그먼트 구조

TCP에서 전송하는 데이터 단위를 **세그먼트(Segment)** 라고 한다. TCP로 전송할 수 있는 최대 페이로드 크기를 **MSS(Maximum Segment Size)** 라고 한다.

TCP 세그먼트 헤더의 주요 필드는 이렇다.

| 필드 | 역할 |
|------|------|
| **송수신지 포트** | 송수신 애플리케이션을 식별하는 포트 번호 |
| **순서 번호** | 세그먼트의 올바른 순서를 보장하기 위한 번호. 데이터의 첫 바이트에 부여 |
| **확인 응답 번호** | 다음으로 수신하기를 기대하는 순서 번호. ACK 플래그가 1일 때 유효 |
| **제어 비트** | 세그먼트의 부가 정보 (ACK, SYN, FIN 등) |
| **윈도우** | 한 번에 수신하고자 하는 데이터의 양 (수신 윈도우 크기) |

**제어 비트(플래그 비트)** 중 핵심 세 가지:

| 플래그 | 역할 |
|--------|------|
| **SYN** | 연결 수립 요청 |
| **ACK** | 세그먼트 수신 승인 |
| **FIN** | 연결 종료 요청 |

**순서 번호와 확인 응답 번호**는 TCP 신뢰성의 핵심이다.

- **순서 번호**: 세그먼트 데이터의 첫 바이트에 부여되는 번호. 연결 수립 시 초기 순서 번호(ISN)는 무작위 값으로 설정된다
- **확인 응답 번호**: 수신한 순서 번호 + 1. "여기까지 받았고, 다음은 이걸 줘"라는 의미

```
MSS가 500바이트, 1900바이트 데이터를 전송한다면:

세그먼트 A: 순서 번호 100  (500바이트)
세그먼트 B: 순서 번호 600  (500바이트)
세그먼트 C: 순서 번호 1100 (500바이트)
세그먼트 D: 순서 번호 1600 (400바이트)
```

### TCP 연결 수립 — 쓰리 웨이 핸드셰이크

TCP 연결을 수립하는 과정을 **쓰리 웨이 핸드셰이크(Three-Way Handshake)** 라고 한다. 세 단계로 이루어진다.

```
호스트 A (클라이언트)              호스트 B (서버)
      │                                │
      │  ① SYN 세그먼트 전송           │
      │ ─────────────────────────────▶ │  "연결 시작합니다"
      │                                │
      │  ② SYN + ACK 세그먼트 전송     │
      │ ◀───────────────────────────── │  "네, 확인했습니다. 연결 시작해요!"
      │                                │
      │  ③ ACK 세그먼트 전송           │
      │ ─────────────────────────────▶ │  "네, 확인했습니다."
      │                                │
     [ESTABLISHED]                [ESTABLISHED]
```

- **액티브 오픈**: 처음 연결을 시작하는 호스트 (주로 클라이언트)
- **패시브 오픈**: 연결 요청을 받아 수립해주는 호스트 (주로 서버)

---

### TCP 연결 종료 — 포 웨이 핸드셰이크

연결을 종료할 때는 **포 웨이 핸드셰이크(Four-Way Handshake)** 를 거친다. 네 단계가 필요한 이유는 양쪽 모두 각자의 연결 종료를 따로 확인해야 하기 때문이다.

```
호스트 A (액티브 클로즈)          호스트 B (패시브 클로즈)
      │                                │
      │  ① FIN 세그먼트 전송           │
      │ ─────────────────────────────▶ │  "연결 끊을게요"
      │                                │
      │  ② ACK 세그먼트 전송           │
      │ ◀───────────────────────────── │  "네, 확인했습니다"
      │                                │
      │  ③ FIN 세그먼트 전송           │
      │ ◀───────────────────────────── │  "이제 연결 끊어요"
      │                                │
      │  ④ ACK 세그먼트 전송           │
      │ ─────────────────────────────▶ │  "네, 확인했습니다"
      │                                │
   [TIME-WAIT]                     [CLOSED]
   (일정 시간 후)
     [CLOSED]
```

### TCP 상태

TCP는 현재 어떤 통신 과정에 있는지 **상태(State)** 를 유지한다. 이런 프로토콜을 **스테이트풀(Stateful)** 이라고 한다.

**연결 전 상태**

| 상태 | 설명 |
|------|------|
| `CLOSED` | 아무런 연결이 없는 상태 |
| `LISTEN` | 연결 대기 상태. SYN 세그먼트를 기다리는 중 |

**연결 수립 상태**

| 상태 | 설명 |
|------|------|
| `SYN-SENT` | 클라이언트가 SYN 전송 후 SYN+ACK를 기다리는 상태 |
| `SYN-RECEIVED` | 서버가 SYN+ACK 전송 후 ACK를 기다리는 상태 |
| `ESTABLISHED` | 연결 확립. 데이터 송수신 가능 |

**연결 종료 상태**

| 상태 | 설명 |
|------|------|
| `FIN-WAIT-1` | FIN 전송 후 ACK를 기다리는 상태 (액티브 클로즈 측) |
| `CLOSE-WAIT` | FIN 수신 후 ACK 전송, 종료 준비 중 (패시브 클로즈 측) |
| `FIN-WAIT-2` | ACK 수신 후 상대 FIN을 기다리는 상태 |
| `LAST-ACK` | FIN 전송 후 마지막 ACK를 기다리는 상태 |
| `TIME-WAIT` | 마지막 ACK 전송 후 일정 시간 대기 (혹시 모를 재전송 대비) |
| `CLOSING` | 양쪽이 동시에 연결 종료를 시도할 때 |

---

### UDP란?

**UDP(User Datagram Protocol)** 는 **비연결형, 신뢰할 수 없는 프로토콜**이다.

TCP와 달리 연결 수립/종료 과정이 없고, 오류 제어도 없다. 그냥 빠르게 보낼 뿐이다. **스테이트리스(Stateless)** 프로토콜이기도 하다.

UDP 데이터그램 헤더는 딱 네 개 필드뿐이라 매우 단순하다.

```
┌──────────────┬──────────────┐
│   송신지 포트  │   수신지 포트  │
├──────────────┼──────────────┤
│     길이      │    체크섬     │
└──────────────┴──────────────┘
```

- **길이**: 헤더를 포함한 UDP 데이터그램의 바이트 수
- **체크섬**: 전송 과정에서 오류(훼손)가 발생했는지 검사

TCP에 비해 헤더가 훨씬 작고 단순하다. 그만큼 오버헤드가 적어 **패킷을 빠르게 처리**할 수 있다.

> **그럼 UDP는 언제 쓸까?**

신뢰성보다 속도가 중요한 상황에 사용한다. 실시간 스트리밍, 온라인 게임, DNS 조회, VoIP(인터넷 전화) 등이 대표적이다. 패킷 하나가 유실되더라도 전체 흐름에 큰 문제가 없는 경우다.

## TCP의 오류, 흐름, 혼잡 제어

TCP가 신뢰성을 보장하는 방법은 세 가지다.

```
TCP 신뢰성 보장 3가지
├── 오류 제어  → 패킷 유실/손상 감지 후 재전송
├── 흐름 제어  → 수신 측이 처리할 수 있는 만큼만 전송
└── 혼잡 제어  → 네트워크 혼잡도에 따라 전송량 조절
```

---

### 오류 제어 — 재전송 기법 (ARQ)

TCP는 두 가지 상황에서 문제가 생겼다고 판단한다.

- **중복된 ACK 세그먼트**를 수신했을 때
- **타임아웃**이 발생했을 때 (재전송 타이머가 만료되면 타임아웃)

이때 **ARQ(Automatic Repeat Request)** 방식으로 재전송해 신뢰성을 확보한다. ARQ에는 세 가지 방식이 있다.

#### ① Stop-and-Wait ARQ

확인(ACK)을 받기 전까지 다음 세그먼트를 보내지 않는다.

```
송신: [세그먼트 1] → 기다림 → ACK 수신 → [세그먼트 2] → 기다림 → ...
```

단순하지만 네트워크 이용 효율이 낮다. 기다리는 시간이 너무 길다.

#### ② Go-Back-N ARQ

**파이프라이닝**을 사용해 여러 세그먼트를 연속으로 전송한다. 도중에 잘못된 세그먼트가 발생하면 **그 세그먼트부터 전부 다시 전송**한다.

```
송신: [1][2][3][4][5] 전송
수신: 1,2 OK, 3 유실
→ 3번부터 다시: [3][4][5] 재전송
```

순서 번호 n에 대한 ACK는 "n번까지 다 받았어"를 의미한다. 이를 **누적 확인 응답(CACK)** 이라고 한다.

**빠른 재전송**: 타이머 만료 전에 **동일한 ACK 세그먼트가 3번** 수신되면 해당 세그먼트를 바로 재전송한다.

#### ③ Selective Repeat ARQ

**파이프라이닝**을 사용하되, 수신 측에서 **제대로 받은 각각의 패킷에 개별로 ACK**를 보낸다. 유실된 세그먼트만 선택적으로 재전송한다.

```
송신: [1][2][3][4][5] 전송
수신: 1 OK, 2 OK, 3 유실, 4 OK, 5 OK
→ 3번만 재전송: [3]
```

이를 **개별 확인 응답(Selective ACK)** 이라고 한다. Go-Back-N보다 효율적이다.

### 흐름 제어 — 슬라이딩 윈도우

파이프라이닝으로 여러 세그먼트를 연속 전송하다 보면 문제가 생긴다. 수신 측이 처리하는 속도보다 송신 측이 더 빨리 보내면 **버퍼 오버플로**가 발생한다.

```
수신 버퍼: 수신된 세그먼트가 애플리케이션에 읽히기 전 임시 저장되는 공간
버퍼 오버플로: 수신 버퍼 크기보다 많은 데이터가 들어왔을 때 발생
```

**흐름 제어(Flow Control)** 는 **송신 호스트가 수신 호스트의 처리 속도를 고려하며 송수신 속도를 균일하게 유지**하는 것이다.

이를 구현하는 방법이 **슬라이딩 윈도우(Sliding Window)** 다.

- **윈도우**: 송신 호스트가 한 번에 파이프라이닝할 수 있는 최대 데이터 양
- 수신 호스트는 TCP 헤더의 **윈도우 필드**를 통해 자신이 받아들일 수 있는 데이터 양을 송신 호스트에게 알려준다
- 송신 측은 이 수신 윈도우 크기에 맞춰 전송량을 조절한다

```
수신 호스트 → "나 지금 3000바이트까지 받을 수 있어" (윈도우 필드에 명시)
송신 호스트 → 3000바이트 이내로만 전송
수신 호스트 → 버퍼 비워지면 "이제 5000바이트까지 가능해" (윈도우 갱신)
```

### 혼잡 제어 — 혼잡 윈도우

흐름 제어는 수신 측의 버퍼를 고려하는 것이었다. 그런데 **네트워크 자체가 혼잡**한 경우도 있다.

**혼잡(Congestion)**: 많은 트래픽으로 인해 패킷의 처리 속도가 늦어지거나 유실될 우려가 있는 네트워크 상황

**혼잡 제어(Congestion Control)** 는 혼잡 정도에 따라 **송신 측이 스스로 전송량을 조절**하는 것이다.

이때 사용하는 개념이 **혼잡 윈도우(Congestion Window)**: 혼잡 없이 전송할 수 있을 것으로 예상되는 데이터 양이다.

혼잡 제어 알고리즘은 세 가지 단계로 이루어진다.

#### ① 느린 시작 (Slow Start)

혼잡 윈도우를 1부터 시작해서 ACK를 받을 때마다 1씩 증가시킨다. 지수적으로 증가한다.

```
1 → 2 → 4 → 8 → 16 → ... (ACK 받을 때마다 2배씩)
```

#### ② 혼잡 회피 (Congestion Avoidance)

혼잡 윈도우가 **느린 시작 임계치(ssthresh)** 에 도달하면, RTT마다 1MSS씩 **선형적으로** 증가시킨다.

```
16 → 17 → 18 → 19 → ... (RTT마다 1씩)
```

혼잡 회피 도중 문제가 생기면:
- **타임아웃 발생**: 혼잡 윈도우를 1로, ssthresh는 현재 윈도우의 절반으로 줄인 뒤 느린 시작 재시작
- **3번 중복 ACK**: 혼잡 윈도우와 ssthresh를 절반으로 줄인 뒤 빠른 회복 수행

#### ③ 빠른 회복 (Fast Recovery)

3번 중복 ACK가 수신됐을 때 빠른 재전송과 함께 수행한다. 느린 시작부터 다시 시작하지 않고, 줄어든 혼잡 윈도우부터 **혼잡 회피**를 다시 시작한다.

**AIMD** 란 이 혼잡 제어의 핵심 원리다.
- **Additive Increase**: 혼잡 없으면 RTT마다 선형적으로 증가
- **Multiplicative Decrease**: 혼잡 감지 시 절반으로 감소

---

### 좀 더 알아보기 — ECN: 명시적 혼잡 알림

기존 혼잡 제어는 패킷 유실로 혼잡을 **사후에** 감지했다. **ECN(Explicit Congestion Notification)** 은 네트워크 중간 장치(라우터)의 도움으로 혼잡을 **사전에** 알리는 방법이다.

동작 방식은 이렇다.

```
① 호스트 A → 라우터 → 호스트 B로 패킷 전달 중
② 라우터가 혼잡 감지 → IP 패킷에 ECN 비트 설정해서 B에게 전달
③ 호스트 B → TCP ACK에 ECE 비트 = 1 설정 → A에게 "혼잡이야" 알림
④ 호스트 A → CWR 비트 = 1 설정 + 혼잡 윈도우 절반으로 줄임
```

패킷을 잃기 전에 미리 혼잡을 알 수 있어서 더 효율적이다.

## 정리

전체 흐름을 한 줄로 정리하면 이렇다.

```
IP의 한계 (신뢰성 없음, 비연결형)
  → 전송 계층이 보완
    → TCP: 연결 수립 + 신뢰성 보장 (오류/흐름/혼잡 제어)
    → UDP: 빠른 속도 (신뢰성 포기)
    → 포트: 어느 애플리케이션으로 전달할지 식별
```

| | 설명 |
|-|------|
| **전송 계층** | IP의 한계를 보완. 신뢰성 + 연결 + 포트 식별 |
| **포트** | 특정 애플리케이션 프로세스를 식별하는 번호 |
| **NPAT** | 포트를 활용해 하나의 공인 IP를 여러 사설 IP가 공유 |
| **TCP** | 연결형, 신뢰성 있는 프로토콜 (스테이트풀) |
| **UDP** | 비연결형, 빠른 프로토콜 (스테이트리스) |
| **쓰리 웨이 핸드셰이크** | TCP 연결 수립 과정 (SYN → SYN+ACK → ACK) |
| **포 웨이 핸드셰이크** | TCP 연결 종료 과정 (FIN → ACK → FIN → ACK) |
| **ARQ** | 재전송 기반 오류 제어 (Stop-and-Wait / Go-Back-N / Selective Repeat) |
| **슬라이딩 윈도우** | 수신 측 처리 속도에 맞춰 전송량 조절 (흐름 제어) |
| **혼잡 윈도우** | 네트워크 혼잡도에 따라 전송량 조절 (혼잡 제어) |
| **ECN** | 라우터가 혼잡을 미리 알려주는 명시적 혼잡 알림 |

전송 계층은 "패킷을 어떻게 믿을 수 있게 전달하느냐"를 다루는 계층이다. 다음 글에서는 응용 계층으로 올라가 DNS와 HTTP에 대해 살펴볼 것이다.
