# 컴퓨터 네트워크 시작하기

## 네트워크, 왜 알아야 할까?

개발자가 되기로 마음먹었다면, 아니 그냥 컴퓨터를 다루는 사람이라면 한 번쯤은 이런 상황을 겪었을 것이다.

> *"왜 갑자기 인터넷이 안 되지?"*  
> *"왜 API 요청이 실패하지?"*  
> *"HTTP랑 HTTPS가 뭐가 다른 거야?"*

이런 물음에 답하려면 **컴퓨터 네트워크**에 대한 기본 지식이 필요하다.

**컴퓨터 네트워크(네트워크)** 란, 여러 장치가 마치 그물처럼 연결되어 정보를 주고받을 수 있는 통신망이다.

그리고 우리가 매일 사용하는 **인터넷**은 이 네트워크들을 연결한 '네트워크의 네트워크'다.

개발자 업무를 생각해보면 네트워크 지식이 왜 필요한지 더 명확해진다.

- **프로그램을 만들 때**: 대부분의 프로그램은 다른 장치와 상호작용하며 실행된다. TCP/UDP, HTTP/HTTPS, DNS, 포트번호 같은 개념은 기본 중의 기본이다.
- **프로그램을 유지보수할 때**: 갑자기 서비스가 안 된다면? 네트워크 어딘가에서 문제가 생긴 것일 수 있다. 어디서 문제가 발생했는지 파악하려면 전체 흐름을 알아야 한다.

---

## 네트워크를 크게 보면 어떻게 생겼을까?

네트워크는 사실 우리가 자료구조에서 배우는 **그래프(Graph)** 와 구조가 같다. 노드(node)와 그 노드를 연결하는 간선(edge)으로 이루어진 구조다.

```
[호스트 A] ─── [스위치] ─── [라우터] ─── [호스트 B]
               (네트워크 장비)         (통신 매체로 연결)
```

### 네트워크를 구성하는 요소들

> **노드에는 어떤 것들이 있을까?**

크게 두 종류로 나뉜다.

**① 호스트 (가장자리 노드)**

정보를 최초로 생성·송신하거나, 최종적으로 수신하는 장치다.

- **서버(Server)**: 파일, 웹, 메일 같은 서비스를 *제공*하는 호스트
- **클라이언트(Client)**: 서비스를 *요청*하고 응답을 받는 호스트

우리가 웹 브라우저로 사이트에 접속할 때, 내 컴퓨터는 클라이언트, 그 사이트를 운영하는 컴퓨터가 서버다.

**② 네트워크 장비 (중간 노드)**

호스트끼리 주고받는 정보가 원하는 곳까지 안전하게 도달하도록 중간에서 돕는 장치다.  
허브, 스위치, 라우터, 공유기 등이 여기에 해당한다.

> **그 노드들은 어떻게 연결되어 있을까?**

각 노드를 연결하는 간선이 바로 **통신 매체**다.

- **유선 매체**: LAN 케이블처럼 물리적으로 연결
- **무선 매체**: Wi-Fi처럼 전파를 이용해 연결

> **그래서 노드끼리 뭘 주고받는 거야?**

**메시지(Message)** 다. 웹 페이지, 파일, 이메일처럼 노드끼리 주고받는 모든 정보를 메시지라고 부른다.

---

### 범위에 따라 네트워크를 나눌 수 있다

| 종류 | 이름 | 범위 예시 |
|------|------|-----------|
| **LAN** | Local Area Network | 가정, 학교, 기업 사무실 내부 |
| **MAN** | Metropolitan Area Network | 도시, 대도시 단위 |
| **CAN** | Campus Area Network | 대학교 캠퍼스, 기업 건물 단위 |
| **WAN** | Wide Area Network | 국가 간, 전 세계 (= 인터넷) |

우리가 매일 쓰는 인터넷은 WAN의 대표적인 예다. 이 WAN은 **ISP(Internet Service Provider)**, 즉 KT, SKT, LG U+ 같은 인터넷 서비스 업체들이 구축하고 관리한다.

---

### 메시지를 어떻게 전달할까? — 교환 방식

> **회선 교환 방식**

마치 전화처럼, 통신 전에 먼저 **전용 회선을 설정**하고 그 회선을 통해 메시지를 주고받는 방식이다.

회선이 연결된 두 호스트는 안정적으로 통신할 수 있지만, 그 회선은 다른 누구도 쓸 수 없다. 사용하지 않는 동안에도 회선이 점유되기 때문에 **이용 효율이 낮다**는 단점이 있다.

> **패킷 교환 방식**

오늘날 인터넷이 사용하는 방식이다. 메시지를 **패킷(packet)** 이라는 작은 단위로 쪼개서 전송한다.

패킷은 정해진 경로 없이 그때그때 최적의 경로로 이동하고, 수신지에서 다시 조립된다.

패킷은 크게 세 부분으로 구성된다.

```
┌──────────────────────────────────┐
│  헤더(Header)                     │  ← 부가 정보 (주소, 프로토콜 정보 등)
├──────────────────────────────────┤
│  페이로드(Payload)                 │  ← 실제 전송하고자 하는 데이터
├──────────────────────────────────┤
│  트레일러(Trailer)                 │  ← 오류 감지 등 부가 정보
└──────────────────────────────────┘
```

> **헤더에 들어가는 주소는 어떤 형태야?**

패킷이 올바른 수신지를 찾아가려면 주소가 필요하다. 대표적으로 두 가지가 있다.

- **IP 주소**: 네트워크 상의 논리적 주소 (ex. `192.168.0.1`)
- **MAC 주소**: 네트워크 장비에 부여된 물리적 주소

> **수신 방식도 다양하다**

| 방식 | 설명 |
|------|------|
| **유니캐스트** | 1:1, 특정 한 수신지에게만 전송 |
| **브로드캐스트** | 1:전체, 같은 네트워크 내 모든 호스트에게 전송 |
| **멀티캐스트** | 1:그룹, 특정 그룹에 속한 호스트들에게 전송 |
| **애니캐스트** | 1:가장 가까운 하나, 그룹 중 가장 가까운 호스트에게 전송 |

## 네트워크를 작게 들여다보면

### 프로토콜 — 약속된 언어

여러 나라 사람이 모인 회의에서 모두가 영어로 말하기로 약속했다면, 서로의 모국어가 달라도 소통이 된다. 네트워크에서 이 역할을 하는 게 바로 **프로토콜(Protocol)** 이다.

> 노드 간 정보를 올바르게 주고받기 위해 합의된 규칙이나 방법

각 프로토콜은 저마다 목적과 특징이 있다.

| 프로토콜 | 역할 |
|---------|------|
| **IP** | 패킷을 수신지까지 전달 |
| **ARP** | IP 주소를 MAC 주소로 변환 |
| **TCP** | 느리지만 신뢰성 높은 전송 |
| **UDP** | 빠르지만 신뢰성 낮은 전송 |
| **HTTP/HTTPS** | 웹 데이터 전송 (HTTPS가 보안상 더 안전) |

프로토콜마다 패킷에 담기는 헤더 정보도 달라진다.

---

### 네트워크 참조 모델 — 계층으로 나누는 이유

네트워크 통신이 이루어지는 과정은 꽤 복잡하다. 이 복잡한 과정을 **계층으로 나누어** 관리하는 것이 **네트워크 참조 모델**이다.

계층으로 나누면 두 가지 장점이 있다.

1. **설계가 쉬워진다**: 각 계층에 맞는 프로토콜과 장비를 독립적으로 설계할 수 있다.
2. **문제 해결이 쉬워진다**: 인터넷이 안 될 때 "물리 케이블 문제인가? IP 문제인가?" 처럼 계층별로 원인을 좁혀나갈 수 있다.

대표적인 두 가지 모델이 있다.

> **OSI 모델 vs TCP/IP 모델**

```
OSI 7계층                   TCP/IP 4계층
─────────────────           ──────────────────
7. 응용 계층   ┐
6. 표현 계층   ├──────────▶  4. 응용 계층
5. 세션 계층  ┘
─────────────────           ──────────────────
4. 전송 계층  ──────────────▶ 3. 전송 계층
─────────────────           ──────────────────
3. 네트워크 계층 ────────────▶ 2. 인터넷 계층
─────────────────           ──────────────────
2. 데이터 링크 계층 ┐
1. 물리 계층        ├────────▶ 1. 네트워크 액세스 계층
                   ┘
```

각 계층이 무슨 역할을 하는지 아래에서 하나씩 살펴보자.

| 계층 (OSI) | 역할 | 키워드 |
|-----------|------|--------|
| **물리** | 비트 신호를 전달 | 케이블, 전파 |
| **데이터 링크** | 같은 네트워크 내 장치 간 통신 | 이더넷, MAC 주소 |
| **네트워크** | 다른 네트워크까지 경로 설정 | IP 주소, 라우터 |
| **전송** | 신뢰성 있는 전송, 흐름/오류 제어 | TCP, UDP, 포트 |
| **세션** | 연결(세션) 생성·유지·종료 | 세션 관리 |
| **표현** | 데이터 변환, 압축, 암호화 | 인코딩, SSL |
| **응용** | 사용자에게 네트워크 서비스 제공 | HTTP, DNS, FTP |

---

### 캡슐화와 역캡슐화 — 택배 포장처럼

데이터를 보낼 때는 각 계층을 거치면서 **헤더를 하나씩 추가(캡슐화)** 하고, 받을 때는 반대로 **헤더를 하나씩 제거(역캡슐화)** 한다.

택배에 비유하면 이렇다. 물건(데이터)을 포장지로 싸고, 그 위에 박스를 씌우고, 운송장을 붙이는 과정이 캡슐화다. 받는 사람은 반대로 운송장을 확인하고, 박스를 열고, 포장지를 뜯어 물건을 꺼낸다.

```
[송신 과정 - 캡슐화]

응용 계층:   [ 데이터 ]
전송 계층:   [ TCP 헤더 | 데이터 ]
네트워크:    [ IP 헤더 | TCP 헤더 | 데이터 ]
데이터링크:  [ 이더넷 헤더 | IP 헤더 | TCP 헤더 | 데이터 | 트레일러 ]
물리 계층:   010101001100...  (비트 신호)
```

```
[수신 과정 - 역캡슐화]

물리 계층:   010101001100...
데이터링크:  이더넷 헤더 확인 후 제거
네트워크:    IP 헤더 확인 후 제거
전송 계층:   TCP 헤더 확인 후 제거
응용 계층:   [ 데이터 ] ← 최종적으로 데이터만 남음
```

### PDU — 계층마다 부르는 이름이 다르다

각 계층에서 다루는 메시지 단위를 **PDU(Protocol Data Unit)** 라고 한다. 같은 데이터라도 어느 계층에서 보느냐에 따라 이름이 달라진다.

| 계층 | PDU 이름 |
|------|---------|
| 응용/표현/세션 | 데이터(Data) |
| 전송 | 세그먼트(TCP) / 데이터그램(UDP) |
| 네트워크 | 패킷(Packet) |
| 데이터 링크 | 프레임(Frame) |
| 물리 | 비트(Bit) |

## 마지막으로 — 네트워크 성능은 어떻게 측정할까?

> **트래픽이 뭐야?**

**트래픽(Traffic)** 은 네트워크 내에서 흐르는 데이터의 양이다. 특정 노드에 트래픽이 과도하게 몰리면 과부하가 생기고 성능이 저하된다. 흔히 말하는 "서버가 터졌다"는 표현이 이 경우다.

> **네트워크 성능을 뭐로 측정해?**

| 지표 | 설명 | 단위 |
|------|------|------|
| **처리율(Throughput)** | 단위 시간당 실제로 전송된 데이터 양 | bps, Mbps, Gbps |
| **대역폭(Bandwidth)** | 단위 시간당 전송할 수 있는 최대 데이터 양 | bps |
| **패킷 손실(Packet Loss)** | 전체 패킷 중 유실된 패킷의 비율 | % |

처리율과 대역폭은 비슷해 보이지만 다르다. 대역폭은 도로의 *차선 수*고, 처리율은 실제로 *달리고 있는 차의 수*라고 생각하면 된다. 차선이 많아도 실제 차가 적으면 처리율은 낮다.

---

이것으로 컴퓨터 네트워크의 기본 구조와 개념을 훑어봤다. 앞으로 TCP/IP, HTTP, DNS 같은 각 프로토콜을 더 깊이 들어가면 개발하면서 마주치는 네트워크 문제들이 훨씬 명확하게 보일 것이다.
