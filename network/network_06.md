# 서비스를 지키는 기술 — 네트워크 심화

지금까지 네트워크가 어떻게 동작하는지를 배웠다. 이번 글에서는 한 발 더 나아가, **실제 서비스를 운영할 때 어떻게 안정적이고 안전하게 유지하는지**를 다룬다.

크게 두 가지 주제다.

```
네트워크 심화
├── 안정성(Stability) — 서버가 죽지 않게
│   ├── 이중화 / 다중화
│   └── 로드 밸런싱
└── 안전성(Security) — 데이터가 새지 않게
    ├── 암호화 (대칭 키 / 공개 키)
    ├── 인증서와 디지털 서명
    └── HTTPS / TLS 핸드셰이크
```

그리고 마지막으로 **무선 네트워크(Wi-Fi)** 까지 살펴본다.

---

## 안정성을 위한 기술

### 가용성이란?

사용자가 폭증해서 서버가 불안정해진다면 어떻게 해야 할까? 먼저 기준이 필요하다.

**안정성**: 특정 기능을 언제든 균일한 성능으로 수행할 수 있는 특성

안정성의 정도를 나타내는 용어가 **가용성(Availability)** 이다. 전체 사용 시간 중에서 실제로 정상 동작한 시간의 비율이다.

```
가용성 = 업타임 / (업타임 + 다운타임)
```

가용성이 높을수록 좋다. 실제로 어느 정도냐면:

| 가용성 | 연간 다운타임 |
|--------|------------|
| 99% (투 나인) | 3.65일 |
| 99.9% (쓰리 나인) | 8.77시간 |
| 99.99% (포 나인) | 52.56분 |
| 99.999% (파이브 나인) | 5.26분 |
| 99.9999% (식스 나인) | 31.56초 |

**고가용성(HA, High Availability)** 은 이처럼 높은 가용성을 목표로 시스템을 설계하는 것을 말한다.

그럼 가용성을 높이려면 어떻게 해야 할까? 핵심은 **결함 감내(Fault Tolerance)** 다. 문제가 발생하더라도 시스템이 계속 기능할 수 있도록 설계하는 것이다.

---

### 이중화 — 예비를 마련하자

가용성을 높이는 가장 기본적인 방법은 **이중화(Redundancy)** 다. 예비(백업)를 마련해두는 것이다.

이중화가 필요한 이유는 **단일 장애점(SPoF, Single Point of Failure)** 때문이다. 시스템 어딘가에 문제가 생겼을 때 시스템 전체가 중단될 수 있는 지점이다.

```
[단일 장애점 예시]
클라이언트들 ──→ 라우터(SPoF) ──→ 서버들
                    ↑
              여기가 죽으면 전체 통신 불가
```

이중화 구성 방식은 두 가지다.

**액티브/스탠바이 (Active/Standby)**

```
[Active]  정상 운영 중
[Standby] 대기 상태

Active에 문제 발생 → Standby가 자동으로 역할 인수 (페일오버)
```

- 한 시스템은 가동, 다른 시스템은 백업 대기
- 성능상 큰 변화는 없음
- **페일오버(Failover)**: 액티브 시스템 문제 발생 시 스탠바이로 자동 전환하는 기능

**액티브/액티브 (Active/Active)**

```
[Active A] + [Active B] 모두 운영 중

A에 문제 발생 → B에 부하가 순간적으로 급증
```

- 두 시스템 모두 가동 상태
- 성능 면에서 유리하지만, 한 쪽이 죽으면 다른 쪽에 부하 급증

예비를 **여러 개** 마련하는 것은 **다중화(Multiplexing)** 라고 한다.

> **티밍(Teaming)과 본딩(Bonding)**
>
> 이중화/다중화의 실제 사례. 윈도우에서는 티밍, 리눅스에서는 본딩이라 부른다.
> 여러 개의 NIC를 이중화/다중화해서 더 성능 좋고 안전한 하나의 인터페이스처럼 보이게 하는 기술이다.

### 로드 밸런싱 — 부하를 고르게 나눠주자

서버를 여러 대 두더라도 모든 요청이 한 서버에만 몰리면 의미가 없다. 그래서 필요한 게 **로드 밸런싱(Load Balancing)** 이다.

**트래픽(Traffic)**: 주어진 시점에 특정 노드를 경유한 패킷의 양

**로드 밸런싱**: 트래픽을 여러 서버에 고르게 분배하는 기술

이를 수행하는 장치가 **로드 밸런서(Load Balancer)** 다.

```
클라이언트 ──→ [인터넷] ──→ [로드 밸런서] ──→ 서버 A
                                        ├──→ 서버 B
                                        └──→ 서버 C
```

로드 밸런서는 두 가지 방식으로 구현할 수 있다.
- **네트워크 장비**: L4 스위치, L7 스위치
- **소프트웨어**: HAProxy, Envoy, Nginx

> **서버 상태는 어떻게 파악해?**

두 가지 방법이 있다.

- **헬스 체크(Health Check)**: 로드 밸런서가 주기적으로 서버 상태를 확인. HTTP, ICMP 등 다양한 프로토콜 활용
- **하트비트(Heartbeat)**: 서버 간에 주기적으로 메시지를 주고받다가 끊겼을 때 문제를 감지

#### 로드 밸런싱 알고리즘

어떤 서버에 부하를 보낼지 결정하는 방법이다.

| 알고리즘 | 설명 |
|---------|------|
| **라운드 로빈** | 서버를 순서대로 돌아가며 부하를 전달 |
| **가중치 라운드 로빈** | 가중치가 높은 서버가 더 많은 부하를 받도록 설정 |
| **최소 연결** | 현재 연결이 가장 적은 서버에 우선 전달 |
| **가중치 최소 연결** | 가중치를 고려한 최소 연결 방식 |
| **기타** | 단순 무작위, 해시 기반, 응답 시간 기반 등 |

---

### 좀 더 알아보기 — 포워드 프록시와 리버스 프록시

실제 클라이언트와 서버 사이에는 수많은 중간 서버들이 존재한다.

**오리진 서버(Origin Server)**: 자원을 직접 생성하고 클라이언트에게 권한 있는 응답을 보낼 수 있는 HTTP 서버

메시지 흐름 방향:
- **인바운드(Inbound)**: 오리진 서버를 향하는 메시지
- **아웃바운드(Outbound)**: 클라이언트를 향하는 메시지

중간 서버에는 두 종류가 있다.

**포워드 프록시 (Forward Proxy)**

```
클라이언트 → [포워드 프록시] → 인터넷 → 서버
```

- 클라이언트가 선택한 메시지 전달 대리자
- 캐시 저장, 클라이언트 암호화, 접근 제한 등의 기능
- 클라이언트 측에 위치

**리버스 프록시 / 게이트웨이 (Reverse Proxy)**

```
클라이언트 → 인터넷 → [리버스 프록시] → 서버
```

- 네트워크 간 통신을 가능케 하는 입구 역할
- 캐시 저장, 로드 밸런서로 동작
- 서버 측에 위치

## 안전성을 위한 기술

안정성(서버가 죽지 않게)을 알아봤다면, 이제는 **안전성(데이터가 새지 않게)** 이다. 암호화와 관련된 내용이다.

---

### 암호화

#### 대칭 키 암호화 (Symmetric Key)

```
암호화 키 = 복호화 키 (같은 키 사용)

송신 측: 평문 + 키 → 암호문
수신 측: 암호문 + 키 → 평문
```

- **장점**: 부하가 적어 빠름
- **단점**: 키가 유출되면 바로 복호화 가능. 키를 어떻게 안전하게 전달하냐가 문제

#### 공개 키 암호화 (비대칭 키 암호화, Asymmetric Key)

```
암호화 키 ≠ 복호화 키 (다른 키 사용)

공개 키(Public Key) : 암호화에 사용. 누구에게나 공개
개인 키(Private Key): 복호화에 사용. 절대 공개하지 않음
```

한 키를 알아도 다른 키를 유추할 수 없어서 안전하다.

- **장점**: 키가 유출되어도 개인 키가 없으면 복호화 불가. 안전
- **단점**: 암호화/복호화에 시간과 부하가 상대적으로 많이 소요

> **세션 키(Session Key)**
>
> 실제 통신에서는 두 방식을 조합해서 사용한다.
> - 처음에 **공개 키 암호화**로 세션 키를 안전하게 교환
> - 이후 실제 데이터 전송은 빠른 **대칭 키(세션 키)** 로 암호화
>
> ```
> A가 세션 키 생성
>   → B의 공개 키로 세션 키 암호화 → B에게 전달
>   → B가 자신의 개인 키로 복호화 → 세션 키 획득
>   → 이후 세션 키로 빠르게 통신
> ```

---

### 인증서와 디지털 서명

공개 키 암호화에는 한 가지 문제가 있다. "상대방이 보내준 공개 키가 진짜인지 어떻게 믿냐"는 것이다.

그래서 등장한 것이 **인증서(공개 키 인증서)** 다.

**인증서**: 공개 키와 그 공개 키의 유효성을 입증하기 위한 전자 문서. **인증기관(CA, Certification Authority)** 에서 발급한다.

인증서에는 **서명 값(Signature)** 이 포함되어 있다.

```
서명 값 = 인증서 내용의 해시 값을 CA의 개인 키로 암호화한 것
```

**디지털 서명(Digital Signature)** 이란 개인 키로 암호화된 메시지를 공개 키로 복호화함으로써 신원을 증명하는 절차다.

인증서 검증 과정은 이렇다.

```
① 서버가 인증서 전달
② 클라이언트가 CA의 공개 키로 서명 값 복호화 → 해시 값 A 획득
③ 클라이언트가 인증서 내용을 직접 해시 함수로 계산 → 해시 값 B 생성
④ A == B 이면 인증서가 유효함 (변조되지 않았음)
```

### HTTPS — HTTP를 안전하게

**SSL(Secure Sockets Layer)** 과 **TLS(Transport Layer Security)** 는 인증과 암호화를 수행하는 프로토콜이다. TLS는 SSL을 계승한 최신 버전이다.

**HTTPS(HTTP over TLS)** 는 HTTP 메시지를 TLS로 암호화해서 안전하게 주고받는 프로토콜이다.

HTTPS 통신은 두 단계로 이루어진다.

```
① TCP 쓰리 웨이 핸드셰이크  ← 연결 수립
② TLS 핸드셰이크            ← 암호화 협상
③ 암호화된 데이터 송수신
```

#### TLS 핸드셰이크 과정

```
클라이언트                              서버
    │                                   │
    │  ① ClientHello                    │
    │ ─────────────────────────────────▶│
    │  (TLS 버전, 암호 스위트, 클라이언트 난수)
    │                                   │
    │  ② ServerHello + Certificate      │
    │ ◀─────────────────────────────────│
    │  (선택한 암호 스위트, 서버 난수, 인증서)
    │                                   │
    │  ③ 인증서 검증 + Finished         │
    │ ─────────────────────────────────▶│
    │                                   │
    │  ④ Finished                       │
    │ ◀─────────────────────────────────│
    │                                   │
    │  ⑤ 암호화된 데이터 송수신 시작     │
```

각 메시지의 역할:

| 메시지 | 역할 |
|--------|------|
| **ClientHello** | 클라이언트가 지원 가능한 TLS 버전, 암호 스위트, 난수 등을 제시 |
| **암호 스위트(Cipher Suite)** | 사용 가능한 암호화 방식과 해시 함수 목록 (예: `TLS_AES_128_GCM_SHA256`) |
| **ServerHello** | 서버가 제시된 정보 중 하나를 선택해 응답 |
| **Certificate** | 서버의 인증서 전달 |
| **CertificateVerify** | 검증을 위한 디지털 서명 전달 |
| **Finished** | 핸드셰이크 완료 신호 |

TLS 핸드셰이크가 끝나면 이후 모든 HTTP 메시지는 협상된 키로 암호화되어 전송된다.

## 무선 네트워크

### 전파와 주파수

무선 통신은 **전파(Radio Wave)** 를 이용한다. 전파는 3kHz ~ 3THz 사이의 진동수를 갖는 전자기파다.

와이파이는 주로 두 가지 주파수 대역을 사용한다.

| 주파수 대역 | 특성 |
|-----------|------|
| **2.4GHz** | 속도는 느리지만 장애물의 영향을 덜 받음. 범위 넓음 |
| **5GHz** | 속도는 빠르지만 장애물에 더 영향받음. 범위 좁음 |

---

### 와이파이와 IEEE 802.11

**IEEE 802.11** 은 오늘날 LAN 환경에서 사용되는 무선 통신 표준이다. 와이파이도 이 표준을 따른다.

| 표준 규격 | 주파수 대역 | 전송 속도 |
|---------|-----------|---------|
| IEEE 802.11a | 5GHz | 54Mbps |
| IEEE 802.11g | 2.4GHz | 54Mbps |
| IEEE 802.11n | 2.4/5GHz | 450Mbps |
| IEEE 802.11ac | 5GHz | 6.9Gbps |
| IEEE 802.11ax | 2.4/5/6GHz | 9.6Gbps |
| IEEE 802.11be | 2.4/5/6GHz | 46.1Gbps |

**채널(Channel)**: 같은 주파수 대역을 더 작은 하위 주파수 대역으로 세분화한 것이다. 같은 2.4GHz라도 여러 채널로 나뉘어 간섭을 줄인다. 자동으로 설정되지만 수동으로 변경도 가능하다.

---

### AP와 서비스 셋

**AP(Access Point)**: 무선 네트워크를 생성하기 위해 필요한 무선 액세스 포인트다.

AP를 경유해 통신이 이루어지는 방식을 **인프라스트럭처 모드(Infrastructure Mode)** 라고 한다.

**서비스 셋(Service Set)**: 무선 네트워크를 이루는 AP와 여러 장치들의 집합이다.

```
[서비스 셋 예시]
SSID A: AP(A) + 기기 3대 → 하나의 무선 LAN
SSID B: AP(B) + 기기 3대 → 또 다른 무선 LAN
```

우리가 와이파이 목록에서 보는 네트워크 이름이 바로 **SSID(Service Set Identifier)** 다.

> **BSS와 ESS**
>
> - **BSS(Basic Service Set)**: 하나의 AP로 구성된 무선 LAN
> - **ESS(Extended Service Set)**: 여러 AP로 구성된 무선 LAN

**비컨 프레임(Beacon Frame)**: AP가 불특정 다수 모두에게 자신의 존재를 알리기 위해 주기적으로 전송하는 브로드캐스트 메시지다. 우리가 와이파이 목록을 볼 수 있는 이유가 바로 이것이다.

## 정리

전체 흐름을 한 줄로 정리하면 이렇다.

```
안정성  → 이중화/다중화로 SPoF 제거
        → 로드 밸런싱으로 트래픽 분산
안전성  → 암호화(대칭 키 + 공개 키)로 데이터 보호
        → 인증서/디지털 서명으로 신원 검증
        → TLS 핸드셰이크로 HTTPS 통신 수립
무선    → IEEE 802.11 표준, 2.4/5GHz, AP와 SSID
```

| | 설명 |
|-|------|
| **가용성** | 전체 시간 중 정상 동작 시간의 비율. 업타임/(업타임+다운타임) |
| **결함 감내** | 문제가 발생해도 계속 기능할 수 있는 능력 |
| **이중화** | 예비(백업)를 마련해 SPoF를 제거하는 방법 |
| **페일오버** | 액티브 시스템 장애 시 스탠바이로 자동 전환 |
| **로드 밸런싱** | 트래픽을 여러 서버에 고르게 분산하는 기술 |
| **헬스 체크** | 로드 밸런서가 서버 상태를 주기적으로 확인하는 것 |
| **대칭 키 암호화** | 암호화/복호화에 같은 키 사용. 빠르지만 키 유출 위험 |
| **공개 키 암호화** | 암호화/복호화에 다른 키 사용. 느리지만 안전 |
| **인증서(CA)** | 공개 키의 유효성을 인증기관이 보증하는 전자 문서 |
| **디지털 서명** | 개인 키로 암호화 → 공개 키로 복호화해 신원 증명 |
| **TLS 핸드셰이크** | HTTPS 통신 전 암호화 방식과 키를 협상하는 과정 |
| **SSID** | 와이파이 네트워크의 이름 (서비스 셋 식별자) |
| **비컨 프레임** | AP가 자신의 존재를 주기적으로 브로드캐스트하는 메시지 |

네트워크 심화는 "어떻게 서비스를 더 안정적이고 안전하게 만드는가"를 다루는 파트다. 이중화와 로드 밸런싱으로 안정성을 확보하고, 암호화와 TLS로 안전성을 보장한다. 실제 서비스를 운영하거나 DevOps 업무를 할 때 직접 마주치게 될 개념들이니 잘 익혀두자.
