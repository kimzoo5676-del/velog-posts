# 사용자와 가장 가까운 계층 — 응용 계층

지금까지 물리 계층, 데이터 링크 계층, 네트워크 계층, 전송 계층을 거쳐왔다. 이제 마지막으로 **응용 계층(Application Layer)** 이다.

응용 계층은 우리가 실제로 사용하는 애플리케이션과 가장 맞닿아 있는 계층이다. 웹 브라우저로 구글에 접속하고, 메시지를 보내고, 영상을 스트리밍하는 것 — 이 모든 행위의 밑바탕에 응용 계층 프로토콜이 있다.

이번 글에서는 응용 계층의 두 핵심 주제를 다룬다.

```
응용 계층
├── DNS  — 도메인 네임을 IP 주소로 변환
└── HTTP — 웹에서 자원을 주고받는 프로토콜
```

---

## DNS와 자원

### 왜 IP 주소 대신 도메인 네임을 쓸까?

인터넷에서 어떤 서버에 접속하려면 결국 IP 주소가 필요하다. 그런데 매번 `142.250.196.142` 같은 숫자를 외워서 접속할 수는 없는 노릇이다.

그래서 사람이 읽기 쉬운 **도메인 네임(Domain Name)** 이 등장했다.

```
IP 주소    → "142.250.196.142"  (사람이 외우기 어렵다)
도메인 네임 → "www.google.com"   (사람이 읽기 쉽다)
```

도메인 네임은 **호스트의 IP 주소와 대응되는 문자열 형태의 식별 정보**다.

그럼 도메인 네임과 IP 주소의 대응 관계는 어디에 저장될까? 두 가지 방법이 있다.

- **hosts 파일**: 내 컴퓨터에 직접 `도메인 → IP` 매핑을 저장한 파일 (개인 전화번호부)
- **네임 서버**: 도메인 네임과 IP 주소를 관리하는 외부 서버 (공용 전화번호부)

그리고 이 네임 서버를 체계적으로 관리하는 시스템 전체를 **DNS(Domain Name System)** 라고 한다. DNS는 단순히 서버 하나가 아니라, **계층적이고 분산된 도메인 네임 관리 체계이자 프로토콜**이다.

---

### 도메인 네임의 계층 구조

도메인 네임은 점(`.`)을 기준으로 계층적으로 구성된다.

```
           루트 도메인 (.)
                │
       ┌────────┼────────┐
      com       kr      edu  ...  ← 최상위 도메인 (TLD)
       │
    example                       ← 2단계 도메인
       │
      www                         ← 3단계 도메인 (호스트 네임)
```

`www.example.com`을 분해하면:

- **루트 도메인**: 최상단에 위치. 보통 생략되지만 `com.`처럼 마지막 점으로 표현 가능
- **최상위 도메인(TLD)**: `com`, `kr`, `org` 등 도메인 네임의 마지막 부분
- **2단계 도메인**: `example`처럼 TLD 바로 아래
- **3단계 도메인**: `www`처럼 그 아래. 일반적으로 호스트 네임이라고도 부름

`www.example.com`처럼 호스트를 완전히 식별할 수 있는 전체 도메인 네임을 **FQDN(Fully-Qualified Domain Name)** 이라고 한다.

---

### 계층적 네임 서버 — IP 주소를 어떻게 알아내?

도메인 네임으로 IP 주소를 알아내는 과정을 **리졸빙(Resolving)** 이라고 한다.

이걸 서버 하나가 전부 처리하면 너무 부하가 크다. 그래서 **계층적으로 분산**해서 관리한다. 리졸빙에 관여하는 네임 서버는 네 종류다.

| 서버 | 역할 |
|------|------|
| **로컬 네임 서버** | 클라이언트가 가장 먼저 찾는 서버. ISP가 제공하거나 구글 `8.8.8.8`, 클라우드플레어 `1.1.1.1` 등 공개 DNS 사용 가능 |
| **루트 네임 서버** | 로컬 네임 서버가 모를 때 질의하는 서버. TLD 네임 서버의 주소를 반환 |
| **TLD 네임 서버** | 최상위 도메인을 관리. 하위 도메인 네임 서버의 주소를 반환 |
| **책임 네임 서버** | 특정 도메인 영역을 직접 관리. 최종 IP 주소를 응답 |

질의 방식은 두 가지다.

**재귀적 질의 (Recursive Query)**

```
클라이언트
  → 로컬 네임 서버에 질의
    → 루트 네임 서버에 질의
      → TLD 네임 서버에 질의
        → 책임 네임 서버에 질의
          ← IP 주소 응답
        ← IP 주소 전달
      ← IP 주소 전달
    ← IP 주소 전달
  ← 클라이언트에게 최종 응답
```

각 서버가 다음 서버에게 질의를 위임하고, 결과를 역순으로 받아 전달하는 방식이다.

**반복적 질의 (Iterative Query)**

```
클라이언트 → 로컬 네임 서버에 질의

로컬 네임 서버 → 루트 네임 서버 질의 → "TLD 서버 주소 알려줄게"
로컬 네임 서버 → TLD 네임 서버 질의  → "책임 서버 주소 알려줄게"
로컬 네임 서버 → 책임 네임 서버 질의 → "IP 주소는 이거야"

로컬 네임 서버 → 클라이언트에게 최종 IP 전달
```

로컬 네임 서버가 직접 각 서버에 질의를 반복하며 최종 결과만 클라이언트에게 전달하는 방식이다.

> **매번 이 과정을 거쳐야 해?**

아니다. **DNS 캐시** 덕분에 그럴 필요가 없다.

- 네임 서버들은 응답받은 결과를 임시로 저장해둔다
- 같은 질의가 오면 캐시에서 바로 응답
- 캐시된 값은 **TTL(Time To Live)** 이 만료되면 삭제

### 자원을 식별하는 URI

DNS가 "어느 서버로 갈지"를 알아냈다면, 이제 "그 서버에서 뭘 달라고 할지"를 표현해야 한다.

네트워크에서 주고받는 모든 대상(HTML, 이미지, 동영상, JSON 등)을 **자원(Resource)** 이라고 한다. 이 자원을 식별하는 통일된 방식을 **URI(Uniform Resource Identifier)** 라고 한다.

URI에는 두 종류가 있다.

| 종류 | 이름 | 설명 |
|------|------|------|
| **URL** | Uniform Resource Locator | **위치** 기반으로 자원을 식별 |
| **URN** | Uniform Resource Name | **이름** 기반으로 자원을 식별. 위치가 바뀌어도 식별 가능하지만 아직 잘 안 씀 |

실무에서는 URL을 훨씬 많이 쓴다. URL 구조는 이렇다.

```
https://www.example.com:8042/over/there?name=ferret#nose
└────┘  └──────────────────┘└──────────┘└──────────┘└──┘
scheme        authority          path        query   fragment
```

| 구성 요소 | 역할 |
|----------|------|
| **scheme** | 사용할 프로토콜. `http://`, `https://` |
| **authority** | 호스트를 특정하는 정보. IP 주소 또는 도메인 네임, 포트 번호 포함 가능 |
| **path** | 자원이 위치한 경로. 슬래시(`/`)를 기준으로 계층적으로 표현 |
| **query** | 추가 데이터 전달. `?`로 시작하는 `키=값` 형태, `&`로 연결 |
| **fragment** | 자원의 특정 부분을 가리킴. HTML에서 특정 섹션으로 이동할 때 사용 |

---

### 좀 더 알아보기 — DNS 레코드 타입

도메인 네임을 구입하면 DNS 레코드를 설정하게 된다. DNS 자원 레코드는 이름, 값, TTL, 유형으로 구성되며, 유형에 따라 역할이 다르다.

| 레코드 유형 | 설명 |
|------------|------|
| **A** | 도메인 네임과 **IPv4** 주소의 대응 관계 |
| **AAAA** | 도메인 네임과 **IPv6** 주소의 대응 관계 |
| **CNAME** | 호스트 네임에 대한 별칭 지정 |
| **NS** | 특정 호스트의 IP 주소를 찾을 수 있는 네임 서버 |
| **MX** | 해당 도메인과 연동된 메일 서버 |

예를 들어 `example.com`에 A 레코드로 `1.2.3.4`를 등록하고, `www.example.com`에 CNAME으로 `example.com`을 지정하면 `www.example.com`으로 접속해도 같은 서버로 연결된다.

## HTTP

### HTTP란?

DNS가 IP를 알려주고 URI로 자원을 식별했다. 이제 실제로 **어떻게 주고받냐**가 필요하다. 그게 바로 **HTTP(Hypertext Transfer Protocol)** 다.

HTTP는 응용 계층에서 정보를 주고받는 데 사용되는 프로토콜로, 오늘날 웹 세상의 기반을 이룬다. 네 가지 특성이 있다.

---

### HTTP의 특성

#### ① 요청-응답 기반 프로토콜

HTTP는 클라이언트가 요청을 보내면 서버가 응답하는 구조로 동작한다. 요청 메시지와 응답 메시지는 형태가 다르다.

```
클라이언트 ──── HTTP 요청 메시지 ────▶ 서버
클라이언트 ◀─── HTTP 응답 메시지 ──── 서버
```

#### ② 미디어 독립적 프로토콜

HTTP는 주고받을 자원의 종류를 제한하지 않는다. HTML이든 이미지든 JSON이든 뭐든 전송할 수 있다. 그저 자원을 주고받을 수단의 역할만 수행한다.

자원의 종류는 **미디어 타입(MIME 타입)** 으로 표현한다. `타입/서브타입` 형식이다.

| 타입 | 서브타입 예시 | 설명 |
|------|-------------|------|
| `text` | `text/html`, `text/css`, `text/javascript` | 텍스트 형식 |
| `image` | `image/png`, `image/jpeg`, `image/gif` | 이미지 형식 |
| `video` | `video/mp4`, `video/webm` | 비디오 형식 |
| `audio` | `audio/wav`, `audio/midi` | 오디오 형식 |
| `application` | `application/json`, `application/pdf` | 바이너리 형식 |
| `multipart` | `multipart/form-data` | 여러 타입 혼합 |

매개변수를 붙여 세부 설정도 가능하다.

```
Content-Type: text/html; charset=UTF-8
```

#### ③ 스테이트리스 프로토콜

HTTP는 **상태를 유지하지 않는(stateless)** 프로토콜이다. 서버는 이전에 어떤 클라이언트가 요청을 보냈는지 기억하지 않는다. 각 HTTP 요청은 독립적으로 처리된다.

> **왜 굳이 상태를 기억하지 않을까?**

- **확장성(Scalability)**: 클라이언트가 특정 서버에 종속되지 않아서 서버를 쉽게 추가할 수 있다
- **견고성(Robustness)**: 서버 하나에 문제가 생겨도 다른 서버로 대체하기 쉽다

그럼 로그인 상태 같은 건 어떻게 유지할까? → 뒤에서 **쿠키**로 해결하는 방법을 설명한다.

#### ④ 지속 연결 프로토콜

초기 HTTP(1.0 이하)는 요청-응답 한 번마다 TCP 연결을 맺고 끊었다. 비효율적이다.

```
[HTTP 1.0]
TCP 연결 → 요청/응답 → TCP 종료
TCP 연결 → 요청/응답 → TCP 종료  ← 매번 연결 비용 발생

[HTTP 1.1 이상]
TCP 연결 → 요청/응답 → 요청/응답 → 요청/응답 → TCP 종료
           └──────────────────────────────┘
                  하나의 연결로 처리
```

HTTP 1.1부터 **지속 연결(Persistent Connection, keep-alive)** 을 공식 지원한다.

### HTTP 메시지 구조

HTTP 메시지는 크게 세 부분으로 구성된다.

```
HTTP 메시지
├── 시작 라인   (요청이면 요청 라인 / 응답이면 상태 라인)
├── 필드 라인   (0개 이상의 HTTP 헤더)
├── (빈 줄)
└── 메시지 본문 (선택적)
```

**요청 메시지의 시작 라인 — 요청 라인**

```
GET /example-page HTTP/1.1
└─┘ └───────────┘ └──────┘
메서드  요청 대상   HTTP 버전
```

- **메서드**: 서버의 자원에 대해 수행할 작업 종류
- **요청 대상**: HTTP 요청을 보낼 서버의 자원 경로 (URI)
- **HTTP 버전**: 사용된 HTTP 버전

**응답 메시지의 시작 라인 — 상태 라인**

```
HTTP/1.1 200 OK
└──────┘ └─┘ └┘
HTTP 버전 상태코드 이유 구문
```

- **상태 코드**: 요청 결과를 나타내는 세 자리 정수
- **이유 구문**: 상태 코드에 대한 문자열 설명

**필드 라인 (HTTP 헤더)**

`헤더이름: 헤더값` 형식으로 구성되며, HTTP 통신에 필요한 부가 정보를 담는다.

```
GET /example-page HTTP/1.1
Host: www.example.com
User-Agent: Mozilla/5.0 ...
Accept: text/html
```

### HTTP 메서드

메서드는 클라이언트가 서버 자원에 대해 수행할 작업의 종류를 나타낸다.

| HTTP 메서드 | 설명 |
|------------|------|
| **GET** | 자원을 조회 |
| **HEAD** | GET과 동일하나, 응답에 메시지 본문 없음 (헤더만) |
| **POST** | 서버로 하여금 특정 작업을 처리하게 함 (주로 자원 생성) |
| **PUT** | 자원을 완전히 대체 (없으면 생성, 있으면 덮어쓰기) |
| **PATCH** | 자원의 일부만 수정 |
| **DELETE** | 자원을 삭제 |
| **CONNECT** | 자원에 대한 양방향 연결 시작 |
| **OPTIONS** | 사용 가능한 메서드 등 통신 옵션 확인 |
| **TRACE** | 자원에 대한 루프백 테스트 수행 |

> **PUT vs PATCH 차이가 헷갈린다면**

```
PUT   → 자원 전체 교체. 안 보낸 필드는 사라짐
PATCH → 일부 수정. 보낸 필드만 바뀌고 나머지는 그대로 유지
```

예를 들어 `{ name: "김주우", age: 25 }` 라는 자원이 있을 때:

```json
// PUT: { "name": "김철수" } 전송
// 결과: { "name": "김철수" }  ← age 사라짐

// PATCH: { "name": "김철수" } 전송
// 결과: { "name": "김철수", "age": 25 }  ← age 유지
```

---

### HTTP 상태 코드

응답 메시지의 상태 코드는 요청 결과를 세 자리 숫자로 표현한다.

```
200번대 → 성공
300번대 → 리다이렉션 (자원이 다른 곳에 있어)
400번대 → 클라이언트 에러 (요청이 잘못됨)
500번대 → 서버 에러 (서버가 문제)
```

#### 200번대 — 성공

| 코드 | 이유 구문 | 설명 |
|------|----------|------|
| 200 | OK | 요청 성공 |
| 201 | Created | 요청 성공, 새로운 자원이 생성됨 |
| 202 | Accepted | 요청을 받았으나 아직 처리 완료 안 됨 |
| 204 | No Content | 요청 성공, 응답 본문에 표시할 데이터 없음 |

#### 300번대 — 리다이렉션

**리다이렉션(Redirection)** 이란 클라이언트가 요청한 자원이 다른 위치에 있을 때, 요청을 그쪽으로 이동시키는 것이다.

**영구적 리다이렉션**: 자원이 완전히 새로운 위치로 이동

| 코드 | 이유 구문 | 설명 |
|------|----------|------|
| 301 | Moved Permanently | 영구 이동. 재요청 메서드 변경될 수 있음 |
| 308 | Permanent Redirect | 영구 이동. 재요청 메서드 변경되지 않음 |

**일시적 리다이렉션**: 자원의 위치가 임시로 변경

| 코드 | 이유 구문 | 설명 |
|------|----------|------|
| 302 | Found | 일시 이동. 재요청 메서드 변경될 수 있음 |
| 303 | See Other | 일시 이동. 재요청 메서드 GET으로 변경 |
| 307 | Temporary Redirect | 일시 이동. 재요청 메서드 변경되지 않음 |

#### 400번대 — 클라이언트 에러

| 코드 | 이유 구문 | 설명 |
|------|----------|------|
| 400 | Bad Request | 클라이언트의 요청이 잘못됨 |
| 401 | Unauthorized | 유효한 인증 없음 |
| 403 | Forbidden | 요청이 서버에 의해 거부됨 (접근 권한 없음) |
| 404 | Not Found | 요청받은 자원을 찾을 수 없음 |
| 405 | Method Not Allowed | 요청한 메서드를 지원하지 않음 |

#### 500번대 — 서버 에러

| 코드 | 이유 구문 | 설명 |
|------|----------|------|
| 500 | Internal Server Error | 서버가 요청을 처리할 수 없음 |
| 502 | Bad Gateway | 중간 서버의 통신 오류 |
| 503 | Service Unavailable | 현재는 처리 불가, 추후 가능할 수도 있음 |

## HTTP 헤더와 HTTP 기반 기술

### HTTP 헤더

메시지의 부가 정보를 담는 `헤더이름: 값` 형식의 필드다. 크게 세 종류로 나뉜다.

#### 요청 시 활용되는 헤더

| 헤더 | 설명 |
|------|------|
| **Host** | 요청을 보낼 호스트 (도메인 네임 + 포트) |
| **User-Agent** | 요청을 보낸 클라이언트 프로그램 정보 (브라우저, OS 등) |
| **Referer** | 클라이언트가 현재 머무르고 있던 URL (어디서 왔는지) |
| **Authorization** | 인증 정보. `Basic` 타입은 `아이디:비밀번호`를 Base64 인코딩한 값 |

#### 응답 시 활용되는 헤더

| 헤더 | 설명 |
|------|------|
| **Server** | 요청을 처리한 서버 소프트웨어 정보 |
| **Allow** | 클라이언트에게 허용된 HTTP 메서드 목록. 405 응답과 함께 사용 |
| **Retry-After** | 자원을 사용할 수 있는 날짜/시각. 503 응답과 함께 사용 |
| **Location** | 자원의 새 위치를 알려줌. 리다이렉션 또는 새 자원 생성 시 사용 |
| **WWW-Authenticate** | 자원 접근에 필요한 인증 방식 설명. 401 응답과 함께 사용 |

#### 요청·응답 모두에서 활용되는 헤더

| 헤더 | 설명 |
|------|------|
| **Date** | 메시지가 생성된 날짜와 시각 |
| **Connection** | 연결 방식 설정 (`keep-alive` / `close`) |
| **Content-Length** | 메시지 본문의 바이트 단위 크기 |
| **Content-Type** | 미디어 타입 (`text/html; charset=UTF-8`) |
| **Content-Language** | 사용된 자연어 (`ko-KR`) |
| **Content-Encoding** | 본문 압축/변환 방식 (`gzip`, `br` 등) |

### 캐시 — 같은 걸 또 받을 필요 없잖아

같은 자원을 매번 서버에서 받아오면 낭비다. 그래서 **응답받은 데이터의 사본을 임시 저장**하는 것이 **캐시(Cache)** 다. 불필요한 대역폭 낭비와 응답 지연을 방지한다.

> **캐시가 오래된 데이터면 어떡해?**

그래서 **캐시 신선도(Cache Freshness)** 라는 개념이 있다. 캐시된 사본이 얼마나 최신 원본 데이터와 유사한지를 나타낸다.

캐시 유효기간을 설정하는 방법은 두 가지다.

```http
# 날짜로 지정
Expires: Tue, 06 Feb 2024 12:00:00 GMT

# 초(second)로 지정
Cache-Control: max-age=1200
```

유효기간이 지난 캐시는 재검사가 필요하다.

- **If-Modified-Since 헤더**: 날짜를 기준으로 원본이 변경됐는지 확인
- **ETag(Entity Tag)**: 자원의 버전 식별자. `If-None-Match` 헤더로 재검사

```http
GET /index.html HTTP/1.1
Host: www.example.com
If-None-Match: "abc"
```

변경이 없으면 서버는 `304 Not Modified`로 응답하고, 클라이언트는 기존 캐시를 그대로 사용한다.

### 쿠키 — stateless인데 어떻게 로그인 상태를 유지해?

HTTP는 스테이트리스 프로토콜이라 서버가 클라이언트를 기억하지 못한다. 근데 쇼핑몰에서 로그인하면 다음 페이지에서도 로그인 상태가 유지된다. 어떻게?

**쿠키(Cookie)** 덕분이다. 쿠키는 **서버가 생성해서 클라이언트에 저장시키는 `이름=값` 형태의 데이터**다.

동작 방식은 이렇다.

```
① 첫 로그인 시
   클라이언트 → POST /login → 서버
   서버 → Set-Cookie: session=abc123 → 클라이언트 (쿠키 저장)

② 이후 요청마다
   클라이언트 → Cookie: session=abc123 → 서버
   서버는 쿠키를 보고 "아, 이 사람이구나" 인식
```

> **쿠키 보안은?**

쿠키는 탈취 위험이 있어서 보안 속성을 설정할 수 있다.

- **Secure**: HTTPS 연결에서만 쿠키 전송
- **HttpOnly**: JavaScript로 쿠키 접근 불가. HTTP 송수신을 통해서만 이용 (XSS 방어)

---

### 콘텐츠 협상과 표현 — 같은 URI인데 다르게 응답할 수 있어?

**콘텐츠 협상(Content Negotiation)** 은 같은 URI에 대해 클라이언트에게 가장 적합한 자원의 형태를 제공하는 메커니즘이다.

예를 들어 한국 사용자에게는 한국어 페이지를, 영미권 사용자에게는 영어 페이지를 제공하는 것이다.

클라이언트는 요청 헤더에 선호하는 형태를 명시한다.

```http
GET /index.html HTTP/1.1
Host: example.com
Accept-Language: ko-KR,ko;q=0.9,en-US;q=0.8,en;q=0.7
Accept: text/html,application/xml;q=0.9,text/plain;q=0.6,*/*;q=0.5
```

`q` 값으로 우선순위를 표현한다. `q`가 없으면 기본값은 1이고, 낮을수록 우선순위가 낮다.

```
ko-KR → q=1.0 (최우선)
ko    → q=0.9
en-US → q=0.8
en    → q=0.7
```

### 좀 더 알아보기 — HTTP/0.9에서 HTTP/3.0까지

HTTP는 버전이 올라가면서 점점 더 효율적이고 빨라졌다.

| 버전 | 주요 특징 |
|------|----------|
| **HTTP/0.9** | GET 메서드만 사용 가능. 헤더 없음. 요청 메시지 한 줄 |
| **HTTP/1.0** | HEAD, POST 등 메서드 추가. 헤더 도입. 지속 연결 미지원 |
| **HTTP/1.1** | 지속 연결 공식 지원. 파이프라이닝, 콘텐츠 협상 등 추가 |
| **HTTP/2.0** | 헤더 압축 전송. 바이너리 기반 메시지. 서버 푸시. 멀티플렉싱으로 HOL 블로킹 완화 |
| **HTTP/3.0** | UDP 기반 QUIC 프로토콜 위에서 동작. 속도 개선 |

> **HOL 블로킹(Head-Of-Line Blocking)이 뭐야?**

같은 큐에서 순차적으로 처리되는 패킷이 있을 때, **첫 번째 패킷 처리가 지연되면 뒤의 패킷들도 전부 지연**되는 문제다.

HTTP/2.0은 **멀티플렉싱(Multiplexing)** 으로 이를 완화했다. 여러 스트림을 병렬로 처리해서 하나가 막혀도 다른 것들이 진행될 수 있게 했다.

HTTP/3.0은 아예 TCP 대신 **UDP 기반의 QUIC 프로토콜**을 사용해 더 빠른 속도를 얻었다.

## 정리

전체 흐름을 한 줄로 표현하면 이렇다.

```
도메인 네임 → DNS로 IP 알아내기 → URI로 자원 식별
→ HTTP로 요청/응답 → 헤더로 부가정보 전달
→ 캐시로 효율화 → 쿠키로 상태 보완 → 콘텐츠 협상으로 최적화
```

| | 설명 |
|-|------|
| **도메인 네임** | IP 주소와 대응되는 문자열 형태의 호스트 식별 정보 |
| **URI / URL / URN** | 자원 식별 방식. URL은 위치 기반, URN은 이름 기반 |
| **DNS** | 도메인 네임을 IP 주소로 변환하는 계층적·분산적 시스템 |
| **리졸빙** | 도메인 네임으로 IP 주소를 알아내는 과정 |
| **DNS 캐시** | 질의 결과를 TTL 동안 저장해 응답 속도 향상 |
| **HTTP** | 웹에서 자원을 요청·응답하는 응용 계층 프로토콜 |
| **스테이트리스** | HTTP는 각 요청을 독립적으로 처리. 상태 미보존 |
| **쿠키** | 스테이트리스를 보완하기 위해 클라이언트에 상태 저장 |
| **캐시** | 응답 사본을 임시 저장해 불필요한 재요청 방지 |
| **콘텐츠 협상** | 같은 URI에서 클라이언트에게 가장 적합한 자원 형태 제공 |

응용 계층은 사용자가 직접 마주하는 프로토콜들이 모여 있는 계층이다. DNS가 주소를 알아내고, HTTP가 그 주소의 서버와 실제로 데이터를 주고받는다. 다음 글에서는 웹 보안을 위한 HTTPS와 TLS 핸드셰이크에 대해 살펴볼 것이다.
